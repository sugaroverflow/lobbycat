## 17:14 UTC — Step 3c: fit-note chat thread

**Did:** Built the conversational layer on top of fit-notes. New server action `sendFitNoteMessage({ companyId, content })` in `src/app/actions.ts` persists the user message immediately (so a downstream model error doesn't lose the turn), then calls Anthropic with a strict alternating user/assistant message list reconstructed from the `fit_note_messages` table — our `'cat'` role maps to `'assistant'`. Grounding (user profile, company facts, latest fit-note body) gets prepended to the FIRST user turn in the list rather than smuggled through `system`, so the Messages API alternation stays valid and grounding rides with the conversation. System prompt is the short conversational sibling of the bullets prompt: short replies (1–4 sentences), specific, no bullets unless asked. `getCompanyBySlug` in `src/lib/queries.ts` now fetches `fitNoteThread` (ordered `asc(createdAt)`) alongside the latest `fitNote`. `FitNotePanel` renders the thread under the bullets with a `you` / `lobbycat` eyebrow pattern (sage-soft left rule for user turns to read as a quote-like signal), and a textarea + send button (⌘/Ctrl+Enter to submit, 2000-char cap, optimistic clear-on-send with draft restoration if the action throws). Thread UI only mounts once a fit-note exists, so empty-state companies don't show a dangling chat box. `tsc --noEmit` clean. Marked step 3 ✅ in `REFACTOR-v0.3.md`. Committed (`05704e5`) and pushed.

**Why this shape:** Grounding-on-first-user-turn (rather than appended to `system`) means the cat's behaviour rules and the company facts stay separable — I can iterate the persona in `system` without invalidating cached prompt prefixes, and the facts live where the conversation lives so future thread-summarisation can pick them up uniformly. Persisting the user message before the API call (instead of in a single transaction with the reply) is the correct durability story for chat: a 5xx from Anthropic shouldn't make Aadi's question vanish from the thread. The thread-only-after-fit-note gate is the small piece that makes the UI honest: there's nothing for the cat to follow up *on* until it's said something first.

## 17:02 UTC — Step 3b: fit-notes regenerate as bullets + caveat

**Did:** Rewrote the Anthropic prompt in `src/app/actions.ts::generateFitNote` to emit a strict format — 3 to 5 short bullets (one per line, `- ` prefix, max ~22 words each, specific not generic) and an optional final `caveat: ` line for honest weaknesses (UK-pigeonhole risk, established-team-vs-build-from-scratch, etc.). Dropped the old "4–6 sentences, single flowing paragraph" instruction along with the "end with a short cited rationale" sign-off from the system prompt, since the caveat line now carries that load. Storage stays the same: the raw text lands in `fitNotes.body`, no schema change. Rewrote `src/components/fit-note-panel.tsx` to parse the body with a small line-walker (`parseFitNote`) that pulls out bullets and any `caveat:` line, then renders the bullets as a `<ul>` with moss-dot markers and serif body type, and the caveat as a hairline-separated muted paragraph with a terracotta `caveat` eyebrow. Legacy paragraph-shaped fit-notes still render: any line that isn't a bullet or caveat falls through as its own bullet, so v0.2 rows look fine until they're regenerated. `tsc --noEmit` clean. Committed (`dfd8828`) and pushed.

**Why this shape:** Keeping the storage as plain text in `fitNotes.body` (rather than a structured `bullets` jsonb column) means 3b ships without a second migration and the chat-thread layer in 3c can sit cleanly on top of `fit_note_messages` without entangling itself with how individual notes are stored. Putting the caveat on a dedicated `caveat:` line — instead of as another bullet — makes the honesty signal visually distinct in the UI (terracotta eyebrow, hairline rule) so Aadi can scan for it without reading the whole note, which is the whole point of the bullets rewrite.

**Next (step 3c):** Chat UI for the fit-note thread. Render `fit_note_messages` under the note, add a textarea + send action that appends to the thread and gets a cat reply grounded in the company facts + the existing note. Probably ~25–30m.

Ran `drizzle-kit generate` — produced `drizzle/0001_loving_talon.sql` and bumped `_journal.json` to two entries. `drizzle-kit push` requires a TTY and we don't have one in this runner, so applied the migration directly against the Neon DB via a small `postgres`-driver script that splits the SQL on `--> statement-breakpoint` and runs each statement under `sql.unsafe`. `tsc --noEmit` clean. Marked step 3 as in-progress in `REFACTOR-v0.3.md` with 3a checked off.

**Why this shape:** Keeping the message table tiny (no `metadata`, no `parentId`, no soft-delete) means 3c can build the chat UI without renaming columns later, and 3b can write rationale bullets into the existing `fitNotes.body` field while the conversation rides on the new table — i.e. fit-note storage stays where it is, and the *thread* is the new layer on top. Role is `text` not an enum so we can add `'system'` later without a schema bump if grounding messages get split out.

**Next (step 3b):** Rewrite the fit-note generator to emit bullets (3–5 short bullets + an optional honesty caveat) instead of a paragraph.
