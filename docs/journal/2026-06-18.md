## 16:50 UTC — Step 3a: `fit_note_messages` table + migration

**Did:** Added the `fit_note_messages` table to `src/db/schema.ts` (id, companyId FK with cascade delete, `role` text holding `'user' | 'cat'`, `content` text, `createdAt` tz-stamp) plus two btree indexes (`company_idx`, `created_idx`) so per-company thread fetches stay cheap and chronological ordering is index-backed. Wired `fitNoteMessagesRelations` (belongs-to company) and added `fitNoteMessages: many(...)` to `companiesRelations`. Exported `FitNoteMessage` / `NewFitNoteMessage` inferred types. Ran `drizzle-kit generate` — produced `drizzle/0001_loving_talon.sql` and bumped `_journal.json` to two entries. `drizzle-kit push` requires a TTY and we don't have one in this runner, so applied the migration directly against the Neon DB via a small `postgres`-driver script that splits the SQL on `--> statement-breakpoint` and runs each statement under `sql.unsafe`. `tsc --noEmit` clean. Marked step 3 as in-progress in `REFACTOR-v0.3.md` with 3a checked off.

**Why this shape:** Keeping the message table tiny (no `metadata`, no `parentId`, no soft-delete) means 3c can build the chat UI without renaming columns later, and 3b can write rationale bullets into the existing `fitNotes.body` field while the conversation rides on the new table — i.e. fit-note storage stays where it is, and the *thread* is the new layer on top. Role is `text` not an enum so we can add `'system'` later without a schema bump if grounding messages get split out.

**Next (step 3b):** Rewrite the fit-note generator to emit bullets (3–5 short bullets + an optional honesty caveat) instead of a paragraph. Update `fit-note-panel.tsx` rendering to a `<ul>` with serif bullets. Will probably touch `src/lib/queries.ts` and `src/app/actions.ts` where regenerate lives.
