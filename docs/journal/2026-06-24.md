# 2026-06-24 — v0.7.2 data: publications population gap

## TL;DR

Fatima flagged the dashboard showing ~6 publication rows across 70 companies. Diagnosis: **two compounding bugs**, both architectural rather than "the pipeline crashed."

1. **The RSS pipeline only knows about 3 companies.** Only 3 of 70 entries in `src/db/seed-data.ts` have `blogRssUrl` / `pressRssUrl` set (DeepMind, Anthropic, OpenAI). The cron at `/api/cron/rss-ingestion` faithfully runs every day at 06:30 UTC — it just has 3 feeds to pull. The cron isn't broken, the company data is sparse.
2. **Glyphie's output never reaches the database.** Glyphie writes per-company `research/feeds/<slug>.json` and a global `research/feed.json`. The dashboard `queries.ts` reads from the `publications` Postgres table. Nothing was syncing JSON → DB. So even on days Glyphie produces rich publication entries (e.g. `research/feeds/anthropic-london.json` has the Fable 5 / Mythos 5 export-control story), the user-facing pages stay empty.

Add to that: the 3 Anthropic/OpenAI/DeepMind RSS URLs in seed-data are themselves partially broken — Glyphie's notes flag `https://www.anthropic.com/news/rss.xml` as 404'd. That's the cherry on top, not the root cause.

## What I shipped

Fix path (b) from the task brief: **build a sync job** that reads `research/feeds/*.json` and upserts into `publications`.

### New code

- `src/lib/feeds-sync/index.ts` — `runFeedsSync({ onlySlugs?, dryRun?, feedsDir? })`. Reads each feed JSON, maps `slug → companyId` from the `companies` table, upserts publications keyed on `(companyId, url)` (the existing unique index). Idempotent. `dryRun` plans without writing. Per-row try/catch so a single bad row doesn't tank the run.
- `src/app/api/cron/feeds-sync/route.ts` — Vercel cron entry, mirrors the auth model and query-param shape of `/api/cron/rss-ingestion` and `/api/cron/ats-feeds`.
- `vercel.json` — added `{ path: "/api/cron/feeds-sync", schedule: "0 9 * * *" }`. 09:00 UTC gives Glyphie's 06:00 run + retry + PR review + merge + Vercel rebuild ~3h of headroom most days. Cron is idempotent so missing a day is harmless — next run catches up.

### Mapping

| Glyphie JSON field | publications column |
| --- | --- |
| `slug` | resolved to `companyId` via `companies.slug` |
| `publications[].title` | `title` |
| `publications[].url` | `url` (conflict key) |
| `publications[].type` | `type` (normalised to blog/press/filing/paper/other) |
| `publications[].date` | `publishedAt` (parsed; null on bad date) |
| `publications[].summary` | `summary` |
| `publications[].topics` | `topics` (defaults to `[]`) |

Upsert uses `onConflictDoUpdate` keyed on `(companyId, url)` so Glyphie's later, better summaries overwrite earlier headline-only stubs.

## What I did NOT do (scope discipline)

- **Did not backfill `blogRssUrl` / `pressRssUrl` for the other 67 companies.** That's editorial / seed-data work and belongs in a follow-up. Honestly, with the JSON sync in place, the RSS pipeline becomes less load-bearing — Glyphie is doing the reading anyway and can be more discerning than a regex-summariser-over-feed.
- **Did not fix the Anthropic RSS 404.** Glyphie flagged it; route to Techie or just delete the dead URL when Fatima sees fit.
- **Did not run the sync against prod DB** from this branch. Once merged + deployed, the 09:00 UTC cron will fire and populate; or someone can hit `/api/cron/feeds-sync?dryRun=1` manually to verify shape first.

## Verification path post-merge

## 19:30 UTC — v0.7.2 data: investigating "some companies aren't scored"

### Brief

Fatima asked: *"on the company page, some are scored and some aren't — why is
that?"* The v0.7.2 plan attributed this to v0.6 having added 30 new companies
to the seed without ever running a scoring pass against them. The expected
remediation was: identify the ~30 unscored companies, score them with Sonnet,
land ~420 (= 70 × 6) rows total. Budget ~$2–4.

### Reality check before scoring

Before spending Anthropic credits I queried Neon to confirm the unscored set.
The brief's `LEFT JOIN ... HAVING COUNT(fs.frame_id) = 0` query returned **0
rows**. A wider audit confirmed:

- `SELECT COUNT(*) FROM companies` → **40** (brief expected 70)
- `SELECT COUNT(*) FROM frame_scores` → **240** (= 40 × 6, every cell present)
- Every one of the 40 companies has `cells = 6` against the 6 active frames
- Newest companies were all seeded `2026-06-20 00:37` and were scored on
  `2026-06-23 16:30–21:45` (see `frame_scores.scored_at`); that earlier
  scoring pass is presumably what populated the cells the brief assumed
  were still empty
- Confidence distribution: 158 low / 73 medium / 9 high (out of 240). High
  share of low-confidence cells likely reads on the page as "thin" rather
  than "missing", but every cell does have a score + rationale + confidence
- Placeholder check — `WHERE rationale ILIKE '%placeholder%' OR ILIKE
  '%model call failed%'` returned one row (`xai-london / Team style`), but
  the match was a false positive: the rationale legitimately uses the word
  "placeholder" to describe xAI's London presence. There are **zero**
  centre-of-scale fallback rows in the DB.

### Decision

No scoring pass run. Spending ~$2–4 of Sonnet credits to overwrite 180 cells
that already have real model outputs would have been wasteful and would have
churned the data Fatima might have been judging when she asked the question.

### What did land in this PR

- `scripts/score-missing.ts` — idempotent maintenance script. Identifies
  companies with zero `frame_scores` rows and scores each (company × frame)
  cell via `rescoreCompanyFrame({ force: true })`. Refuses to run unless
  `ANTHROPIC_API_KEY` is set (no placeholder writes), supports `DRY_RUN=1`
  for safe inspection, batches at `SCORE_BATCH` (default 4), writes a CSV
  summary to `docs/score-missing-v0.7.2.csv`. As of today it no-ops because
  the unscored set is empty — but it is the right tool the next time a seed
  expansion forgets to rescore.
- This journal entry.
