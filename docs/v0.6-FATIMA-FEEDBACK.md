## 2026-06-23 15:43 UTC · Auto-rescore on frame definition edit + animated cat

**Decision:** Flip the Step 6 assumption. Frame definition edits MUST trigger
auto-rescore — this is what makes the engine feel live. Cost concerns are
secondary to the product feeling alive.

**Implementation requirements:**

1. **`updateFrame` action** (in `src/app/actions.ts` or wherever Step 6 put it)
   must fire `rescoreCompanyFrame` for every company × this frame, in the
   background (don't block the save). Server action returns immediately;
   the rescore happens in the worker.

2. **Detection of meaningful change**: Compare old vs new frame fields. If
   ANY of `name`, `lowLabel`, `highLabel`, `lowDescription`, `highDescription`,
   `description` changed in a way that affects the prompt — rescore. Pure
   formatting / whitespace-only edits don't trigger.

3. **Animated cat during rescoring**:
   - When rescoring is in flight (cells are stale and the worker is running),
     show an animated cat somewhere visible — probably the welcome card
     area on `/`, or a floating indicator. The cat should look like it's
     *doing something*: tail flicking, ears twitching, a tiny "..." bubble,
     a typewriter-style "the cat is re-reading..." line that cycles through
     the existing `rescoring[]` quotes.
   - Use the existing pixel cat asset (`public/cat/lobbycat.png` or its v0.6
     successor) — animate via CSS keyframes (tail wiggle, blink) or by
     showing 2–3 different sprite frames if available. If a sprite-sheet
     doesn't exist yet, the simplest path is a CSS-only animation on the
     single sprite: a 2-frame "blink" + a sway. Keep it lightweight.
   - The cat disappears when no rescores are in flight.

4. **Cost protection (still important)**:
   - Debounce: the auto-rescore fires only after the user has stopped editing
     for 2 seconds. Save still happens immediately on blur; rescore waits
     2s.
   - If the user makes another edit within those 2s, reset the timer.
   - The rescore worker dedupes: if a (company × frame) cell already has
     a pending rescore for this `frame_version`, don't queue another.

5. **Telemetry**: log frame-edit rescore events to the journal in the
   next chunk that touches it, so we can see if the live-rescore behaviour
   is firing as expected and how much it costs in practice.

**Why this trade-off works:**
- 2-second debounce solves the typo problem (typo → fix within 2s → only one
  rescore at the end)
- The animated cat makes the latency feel like personality, not lag
- Aadi gets the "I edited the frame and watched the field rearrange itself"
  feel that the original product sentence promised
