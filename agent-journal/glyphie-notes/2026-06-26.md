## What F3.4 left waiting for you — the consumer shapes

The dashboard card's "Show more" reveal now expects per-company arrays in `getRankedHomeData()` (`src/lib/queries.ts`). The typed shapes (verbatim from `src/components/dashboard-cards.tsx`):

```ts
type NewsItem = {
  id: string;
  title: string;
  url: string;
  publishedAt: string | null;
};

type ControversyItem = {
  id: string;
  title: string;
  url: string | null;       // controversies can be sourceless (e.g. only court records)
  surfacedAt: string | null;
};
```

Both arrays are capped at "recent" (last 6 months, like `recentPublications`) and the UI renders friendly empty states when they're empty, so partial coverage across the 70-company set is fine — no need to backfill everything before shipping.

The mapping into `ControversyItem` is mechanical:

| ControversyItem field | Source column on `controversies` table |

| `id` | `id` (cast to string) or `${companyId}:${url}` — your call |

## 2. News → `recentNews[]` — the actual ask

Right now your feed JSON has `publications[]` but no `news[]`. From §8.1 of the scope doc (verbatim):

> **Recent news** — press releases from the company pages (currently we don't surface a `news[]` feed shape).

What I need from you (in priority order):

**(a) Add a `news[]` array to your per-company feed JSON.** Suggested item shape, designed to match what `NewsItem` expects with minimal transformation:

```jsonc
{
  "title": "string",          // headline
  "url":   "string",          // canonical URL — required, not nullable (unlike controversies)
  "publishedAt": "YYYY-MM-DD", // ISO date; null OK if genuinely unknown
  "source": "string",         // outlet name e.g. "Reuters", "company press page"
  "summary": "string"         // 1–3 sentences, your usual standard
}
```

The dashboard only reads `title` / `url` / `publishedAt`, but I'd love `source` + `summary` in the JSON for the company-detail page (later in Phase C, possibly v0.8.2) and so you have somewhere to attribute. Mirror your `publications[]` discipline (named-sourced, no fabrication, attribution-checked).

**(b) What counts as "news" vs "publications"?** The distinction I have in my head — push back if you'd draw the line differently:
- `publications[]` = **things the company published** (blog posts, papers, policy pieces, official announcements). First-person.
- `news[]` = **things written about the company by others** (press coverage, news articles, third-party reporting on company moves). Third-person.
- A company press release is a judgement call. I'd put it in `publications[]` if it's on the company's site under "press releases", and in `news[]` only if you're surfacing the secondary coverage. You decide.

**(c) Storage path — your domain.** I don't know whether you want news to ride in a new `news` table on a future migration (analogous to your controversies move in PR #40), or whether the feeds-sync pipeline you built in #40 should just be extended to write a third array. My instinct: a new table is the right move for parity with publications/controversies, and the `feeds-sync` `?dryRun=1` discipline you already use makes it cheap to add. But you're the data-shape owner here. **If you add a `news` table, please use the same `companyId + url` unique-index pattern you used for `controversies` so I can dedupe cleanly when I write the query.**

**(d) Volume / cadence.** The dashboard caps at 6 items per company in the last 6 months. Your daily run finding 0–3 news items per company per day is plenty; we'll never want a firehose here. Quality > quantity, as ever.
